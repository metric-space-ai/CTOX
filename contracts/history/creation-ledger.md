# Creation Ledger

## 2026-04-01
- CTOX chat-turn execution now has an explicit turn-engine layer instead of leaving planning, compaction assessment, continuity refresh accounting, and end-of-turn outcome shaping implicit inside one chat function.
- The live chat path still runs through `src/inference/chat.rs` plus `src/lcm.rs`, but it now emits a clearer turn plan, compaction guard, context-selection summary, and structured turn outcome so context pressure and continuity refresh behavior are observable as first-class runtime state rather than incidental log text.
- CTOX's mission loop now also carries a first dedicated mission governor: when the service sees the same blocker recur without fresh progress and context-health warnings confirm loop pressure, it now queues a repair/replan slice instead of relying only on normal continuation or blocked follow-up retries.
- CTOX now treats interactive browser work as a first-class fourth web path beside current-search, source-reading, and durable scraping: the workspace gained a native `ctox browser` bridge, a dedicated `interactive-browser` system skill, and chat-runtime `js_repl` wiring that can resolve Playwright from a standard runtime reference workspace under `runtime/browser/interactive-reference/`.
- The new browser path is intentionally specialist rather than default. CTOX should prefer `WebSearch` for discovery, `WebRead` for concrete source evidence, and `WebScrape` for recurring extraction, and only reach for the live browser path when DOM behavior, JavaScript state, auth/session state, or visual evidence is actually the source of truth.

## 2026-03-30
- CTOX skill presentation was reorganized around four explicit operator-facing classes instead of mixed path-derived labels: `CTOX Core`, `Codex Core`, `Installed Packs`, and `Personal`.
- The TUI skill catalog now derives those classes from explicit metadata when present and otherwise falls back to the new repo/global directory conventions, removing the previous confusing `ctox system` / `codex system` / `repo custom` wording.

## 2026-03-23
- CTOX was refactored so its conversational loop can run as a detached local background service started with `ctox start` and stopped with `ctox stop`.
- The TUI was changed from owning the loop itself to attaching to the running service, with loop control exposed in Settings.
- Linux installs now provision a persistent `systemd --user` `ctox.service`, and `ctox start` or `ctox stop` now map to enable-and-start vs stop-and-disable semantics on hosts where that unit is installed.
- CTOX now keeps a parallel local sidecar set alive beside the chat backend: one embeddings server, one STT server, one TTS server, plus the local responses proxy.
- CTOX now has a first-class `schedule` channel: cron-style recurring jobs live in SQLite, emit normal inbound messages, and are routed through the same serial coil and service loop as TUI, Jami, and email.
- Shared auxiliary GPUs are no longer treated as all-or-nothing for chat inference: CTOX now reduces chat layer placement on GPUs that also host embedding, STT, or TTS backends, and drops out of even NCCL sharding when that asymmetric placement is required.
- The short-lived Qwen3-TTS `vllm-omni` detour was removed again; CTOX is back to a single `mistralrs` / `ctox-vllm-serve` runtime path for chat, embeddings, STT, and the current native TTS backend.

## 2026-03-24
- CTOX gained explicit `plan` and `follow-up` tools around Codex: longer owner requests can now be drafted into compact plan artifacts, optionally persisted as tracked steps, and evaluated at turn end without changing the Codex core.
- Plan steps now carry explicit lifecycle state (`pending`, `queued`, `completed`, `blocked`, `failed`) so blocked or failed long-running work is no longer hidden inside transient chat output.
- CTOX now ships a bundled `plan-orchestrator` system skill with explicit contracts for when long-running work must be persisted instead of handled as an in-memory todo list.
- CTOX now also exposes an explicit `queue` tool and `queue-orchestrator` skill: Codex can read and edit the durable inbound task queue itself instead of relying on hidden wrapper logic for follow-up work.
- The native `mistralrs` TTS path was switched from the temporary Dia backend to `Qwen/Qwen3-TTS-12Hz-0.6B-Base`, and the chat launcher now prefers live per-GPU auxiliary VRAM measurements when rebalancing LLM layer placement on shared GPUs.

## 2026-03-25
- CTOX gained eight bundled admin system skills for infrastructure discovery, reliability work, controlled change execution, security posture review, recovery assurance, incident response, automation engineering, and operations insight.
- These admin skills were integrated as additional skill-layer operating knowledge on top of the existing CTOX queue, plan, schedule, follow-up, and service-loop substrate rather than as a second autonomous execution loop.
- The host-observability side of CTOX's admin surface is now anchored in concrete local tooling such as `htop`, `btop`, `top`, `ps`, `vmstat`, `iostat`, `ss`, `journalctl`, `systemctl`, `df`, `du`, `curl`, and `nvidia-smi`.
- The eight admin skills now share one SQLite evidence kernel (`discovery_run`, `discovery_capture`, `discovery_entity`, `discovery_relation`, `discovery_evidence`) keyed by `skill_key`, so discovery, reliability, incident, change, security, recovery, automation, and insight runs persist into one interoperable local source of truth.
- The bundled admin helper scripts are now explicitly open, inspectable resources for Codex rather than hidden black-box function calls; each skill can reuse, patch, or bypass them while still writing back into the same SQLite layer.
- CTOX now also has a canonical ops-template skill that defines the shared invariants, section layout, and refinement escalation ladder for the eight ops skills, so later refinement can adapt helpers and editable sectors without silently breaking family consistency or the shared SQLite kernel.
- The ops skill family now carries an explicit operator-feedback contract: every user-facing answer must say whether the result is proposed, prepared, executed, or blocked, and must separate internal persistence details from the operator-visible outcome.
- CTOX now also injects a dedicated `queue-cleanup` system-skill turn when the in-memory service queue crosses a pressure threshold, so the next execution slice can diagnose and contain queue floods before normal work continues.
- CTOX mail routing now treats only the configured owner email address as an instruction-bearing inbound email source; non-owner email is blocked from the execution queue, and the service now syncs configured email inboxes directly into the shared inbound path.

## 2026-03-26
- CTOX auxiliary model policy now distinguishes explicit `[GPU]` and `[CPU]` variants in the TUI and proxy configuration instead of assuming every embedding, STT, and TTS sidecar has CUDA available.
- The embedding sidecar can now keep `Qwen/Qwen3-Embedding-0.6B` on a CPU-only mistralrs path when a host has no GPU.
- The CPU STT fallback now switches to a multilingual `faster-whisper` path, and the CPU TTS fallback is now standardized on Speaches-backed Piper voices so CTOX installs without GPUs can still transcribe and synthesize speech locally in languages such as German and French without carrying multiple CPU TTS families.
- Owner-facing communication is now constrained more tightly: email replies on existing threads must preserve the thread subject, and CTOX should not promise future high-impact work without also creating an explicit durable follow-up or review record in queue or plan state.
- Owner-visible turns now carry a runtime follow-up safety net: if CTOX leaves a task explicitly open or a turn fails before closure, CTOX creates a durable queue continuation/recovery task instead of silently dropping continuity.
- The ops skill family now hardens the same operator-facing contract across discovery, reliability, incident, change, security, recovery, automation, and insight: every reply must separate `proposed`, `prepared`, `executed`, and `blocked`, and open multi-step work must point to durable next-work state instead of vague promises.
- Owner communication now explicitly depends on both thread continuity and the recent owner-relevant cross-channel communication state, so email replies should no longer act as if only the latest inbound line exists.
- Blocked owner-visible work now has to name the exact missing inputs or approvals, tell the owner how to unblock it, and gets an automatic recurring review schedule instead of relying on long-running waits or hopeful timeouts.
- CTOX now has the first delivery-family layer alongside the ops-family layer: a canonical delivery template plus `service-deployment` and `secret-management`, so local installs can be treated as bounded deployment work with credential classification instead of being mistaken for external integrations or vague blocker mail.
- The synchronous continuity refresh was removed from the pre-reply critical path of chat turns; CTOX now answers first and refreshes continuity best-effort afterward so long-running delivery turns do not appear hung before they can report completion.
- Email authorization is now role-based instead of owner-only: the configured domain may contact CTOX for support and account-management work, explicit admin mail profiles may authorize admin work, and sudo authority is tracked separately from plain admin rights; secret-bearing input still stays TUI-only.
- 2026-03-26: The delivery family now treats verification as a separate layered acceptance step instead of conflating “service started” with “deployment succeeded”; deployments that only reach a web UI or listener but fail authenticated/admin verification must stay `needs_repair` or `blocked` until the higher verification layer passes.
- CTOX chat turns now cap how much raw recent conversation is rendered into a live prompt and explicitly prefer compact recent summaries plus the newest unsummarized items, so long operator histories no longer bloat Codex turns into avoidable timeouts.
- CTOX runtime failures such as a `codex-exec` timeout are now surfaced back to the operator as a shaped `blocked` assistant reply and immediately paired with a durable recovery follow-up, instead of leaving the turn silent or leaking raw runtime event streams into user-facing status.
- CTOX now distinguishes a normal base chat model from an optional stronger boost model; the proxy can grant temporary boost leases with explicit reasons and automatic expiry, and the TUI now surfaces the base model, active model, and boost state instead of hiding model escalation.
- Owner communication now treats historical context as an explicit capability rather than a prompt dump: inbound mail wrappers point CTOX to active communication search tools (`ctox channel history`, `ctox channel search`, `ctox lcm-grep`), and the communication skill family now requires reconstructing relevant prior state before replying.
- CTOX now has a dedicated `self-improving-review` skill beside refinement: skill mutations must be reviewed against evidence, written into a dedicated skill-improvement ledger, and only then reported outward as successful self-optimization.
- CTOX now also exposes a structured `ctox channel context` view that groups thread state, cross-channel related messages, current blockers, and still-open owner questions into one communication-state snapshot instead of forcing mail replies to infer continuity from raw recent snippets.
- Chat compaction is now preset-owned instead of split between preset logic and a free-floating TUI slider: each chat preset carries both its percentage trigger and its absolute minimum compaction floor, and the TUI context bar now renders the real compaction threshold rather than recomputing an approximate percentage-only marker.
- CTOX skill evolution now has an explicit lifecycle layer: new or changed skills can move through states such as `draft`, `candidate`, `promoted`, `deprecated`, and `retired`, and those transitions are tracked in a dedicated skill-lifecycle ledger instead of being implicit side effects of file edits.
- Jami onboarding no longer disappears completely behind a runtime failure: the adapter now exposes a `doctor` view with concrete host prerequisites, and the TUI renders a configured Jami QR target together with repair hints when DBus/runtime resolution is broken instead of replacing the QR entirely with a generic error block.
- The settings TUI now separates `General` and `Model` concerns for real, throttles Jami QR/runtime probing so the interface stays responsive, keeps the QR/details block scoped to the actual Jami settings focus, and treats base-vs-boost model selection as first-class settings instead of burying them behind stale snapshots.
- CTOX now emulates Responses `web_search` locally in the proxy for non-OpenAI backends: when the request enables web search, the proxy executes a reviewed external search, injects compact result context into the turn, and emits a synthetic `web_search_call` item back on the Responses surface instead of silently dropping the tool.

## 2026-03-27
- CTOX no longer presents its local inference layer as a `vllm-serve`/`mistralrs` composite in the active project surface. The runtime was renamed around `engine`, `gateway`, and `runtime` terms, the active inference modules moved under `src/inference/`, the engine tree now lives under `engine/candle/`, and the external launcher/binary surface now points to `ctox-engine` so the current architecture reads as one CTOX-owned Candle-based engine stack instead of a pile of inherited project names.
- The Candle engine workspace itself was then re-cut by role instead of upstream project names: `mistralrs-*` directories were replaced by `cli`, `server`, `core`, `sdk`, `vision`, `quant`, `paged-attn`, `audio`, `mcp`, and `macros`, while Cargo package names moved to a `ctox-engine-*` scheme and temporary compatibility stayed only at the Rust crate-id layer.
- The local `web_search` runtime now keeps a real per-request session plus a persisted page cache, so `search`, opened-page extraction, and `find_in_page` evidence reuse one normalized source-reading path instead of repeatedly refetching pages or reconstructing evidence heuristically each time.
- PDF source reading inside local `web_search` is now page-aware and query-guided: CTOX keeps per-page PDF evidence, reloads hinted pages when the query asks for specific page ranges, and surfaces page-labeled excerpts/matches instead of flattening every PDF into one early-page blob.
- The local `web_search` reader is now explicitly adapter-based instead of pretending one HTML heuristic fits every site: CTOX separates GitHub, docs-style sites, generic HTML, and plain-text paths, and keeps platform-specific extraction inside those reviewed adapters so it can move closer to OpenAI-style web-search behavior without collapsing into one giant scraper branch.
- The local `web_search` reader now also has explicit knowledge- and news-oriented paths, and important high-signal sources like Wikipedia and arXiv can be normalized through platform-specific metadata/extract endpoints instead of relying only on brittle page HTML scraping.
- The GitHub branch of local `web_search` is now API-first for repo, tree, and blob pages: CTOX fetches structured repository metadata plus query-guided supplemental files like `src/main.rs` before falling back to page HTML, so repo-root questions can surface deeper operational evidence closer to OpenAI-style GitHub exploration.
- CTOX now also has a first dedicated `universal-scraping` system skill plus a hybrid scrape registry model: repo-managed skill resources stay stable, target-specific generated scripts live under `runtime/scraping/targets/<target_key>/`, and mutable scrape metadata such as revisions, promoted templates, runs, and artifacts persist in `runtime/ctox_scraping.db` instead of being scattered across ad hoc files.
- The scrape model deliberately ports the strongest management pattern from NinjaWorkflow_Services without copying its whole service stack: target-level script revisions stay separate from promoted provider-family templates, and recurring scrape work is expected to flow through normal `ctox schedule` state rather than through a second hidden scheduler.
- CTOX now also exposes a native `ctox scrape` CLI bridge so scrape targets, script revisions, execution runs, drift classification, and repair follow-up can stay inside the existing CTOX queue/schedule/service loop instead of depending on a second agent runtime; drift-like failures emit a repair bundle plus a normal CTOX queue task, while temporary portal outages are recorded without triggering script rewrites.
- CTOX now carries the first native Voxtral-TTS ingestion path inside the Candle speech stack itself: the engine can resolve `params.json`, `tekken.json`, preset voice embeddings, and Voxtral-specific request shaping as a separate speech family instead of pretending every GPU TTS model is Qwen-shaped. The actual semantic/acoustic generator and audio-tokenizer decode path are still incomplete and remain explicit follow-on port work.
- The temporary Python scrape helpers were retired from the active CTOX path: scrape registry management, execution, template learning, and drift-triggered repair enqueueing now live natively in Rust under `ctox scrape`, while generated target extractors are expected to remain normal JavaScript/TypeScript or shell runtime files under `runtime/scraping/targets/`.
- The scrape runtime now also materializes a canonical latest-record state per target instead of stopping at per-run artifacts: successful runs compute keyed inserts/updates/deletes into `scrape_record_latest`, write target-local `state/latest_records.json` and `state/latest_summary.json`, expose that state through `ctox scrape show-latest`, and include the last successful materialized context in repair bundles so future drift repair no longer starts from raw stdout alone.
- The scrape runtime now also treats queryability as a first-class target artifact instead of an afterthought: each target gets a default runtime API contract plus editable semantic and LLM-enrichment templates under `runtime/scraping/targets/<target_key>/api/`, the CTOX service exposes generic `api`, `records`, `latest`, and `semantic` endpoints per target, and semantic retrieval reuses the configured embedding sidecar through a target-level embedding cache instead of hardwiring one scraper-specific search API.
- The scrape runtime now also executes target-local LLM enrichment as part of the normal successful scrape path instead of leaving it as documentation only: when `llm_enrichment_template.json` is enabled, CTOX calls the existing responses runtime, merges returned classification/extraction/summary updates into each record before latest-state materialization, records enrichment reports as run artifacts, and preserves edited target-local templates across later manifest refreshes instead of overwriting them.
- Multi-source scraping is now a first-class CTOX target contract instead of a hidden convention inside one script: `config.sources[]` is normalized into a target-local source graph, each source gets its own runtime folder under `runtime/scraping/targets/<target_key>/sources/`, source-local extractor modules can be revisioned natively with `ctox scrape register-source-module`, and run manifests now carry the source/module context needed to repair or customize one upstream without pretending the whole scraper is one opaque blob.

## 2026-04-01 - Detached runtime now ignores session hangups

- Hardened detached service and backend spawns so CTOX keeps running after the launching shell or SSH session disappears.
- `service` and `inference::supervisor` now ignore `SIGHUP`/`SIGPIPE` after `setsid()` during detached startup.
- The foreground CTOX service loop now re-binds and continues if the HTTP accept loop ends unexpectedly instead of letting the process die silently.
- This is an operational fix for long-running mission loops and remote benchmark sessions, not a change in CTOX identity or governance.
- CTOX now has a first native completion-review gate in the service/follow-up path: risky or closure-claiming slices can trigger a second read-only `codex exec` review pass, and the service no longer lets such slices fall through to `done` purely because the implementation reply sounded finished if review is missing or non-pass.
- CTOX now persists mission-facing verification state in its LCM runtime instead of treating review as a transient side note: risky slices record `verification_runs`, update durable `mission_claims`, and drive follow-up closure from open closure-blocking claims so long-running code, ops, and mission work no longer depend only on the agent's latest completion prose.
- The Airbnb long-run bench now defaults its managed service launcher to a plain detached process instead of `systemd-run --user`; transient user units remain opt-in via `CTOX_BENCH_USE_SYSTEMD=1`, because host-local systemd cgroup lifecycles were tearing down benchmark services too aggressively during heavy model startup.
- The Airbnb long-run bench no longer hardcodes `Qwen/Qwen3.5-35B-A3B` into every prepared runtime root; it now accepts an explicit model choice and defaults to `openai/gpt-oss-20b`, so benchmark results can reflect the intended local execution policy instead of a hidden harness override.
- The backend supervisor no longer uses `fuser -k` for port cleanup after we confirmed via `strace` on `100.96.1.7` that the host's `fuser` path issued `kill(0, SIGTERM)` and could terminate the CTOX service itself; listener cleanup is now done by signaling concrete PIDs only.
- The backend supervisor also no longer tries to stop managed backends via negative-PID process-group signals; after a second `strace` on `100.96.1.7` showed `/usr/bin/kill -TERM -<pid>` cascading into another self-destructive group kill, CTOX now terminates only the concrete managed PID and escalates to `SIGKILL` on that PID if needed.
- The Airbnb long-run bench now tears down the full local runtime on reset instead of only the foreground service wrapper: it tries `ctox stop` before killing the managed service, explicitly removes proxy/backend pid-file processes, and clears listeners on the configured proxy/backend ports so stale sidecars cannot survive a reset and keep routing to old ports like the `22434 -> 1235` proxy failure seen on `100.96.1.7`.
- The Airbnb long-run bench setup also now strips stale derived proxy/runtime fields like `CTOX_UPSTREAM_BASE_URL` before writing the per-root `engine.env`, because a preserved Qwen-era upstream URL was enough to make a freshly spawned `gpt-oss-20b` proxy still relay to `127.0.0.1:1235` on `100.96.1.7`.
- The local responses proxy now also respects the active per-root chat port override when it boots from runtime env: for local models it prefers `CTOX_ENGINE_PORT` over the static model-family default, which fixes the host1 case where a healthy `gpt-oss-20b` backend bound `2235` but the proxy still relayed to `127.0.0.1:1234`.
- CTOX now has an explicit auxiliary-backend disable path instead of always booting embedding/STT/TTS sidecars from default selections: the supervisor and proxy honor runtime flags like `CTOX_DISABLE_AUXILIARY_BACKENDS=1`, disabled auxiliary routes return a clean `503`, and the Airbnb long-run bench now sets that chat-only mode by default so coding benchmarks do not burn VRAM on unrelated sidecars.
- The GPT-OSS harmony bridge no longer holds Codex's `/v1/responses` SSE client completely silent while it buffers the full completion behind the proxy: CTOX now opens the downstream event stream immediately, emits keepalive comments while the harmony roundtrip runs in the background, and only appends the final translated Responses payload once the buffered completion is ready. This is an operational relay hardening for long turns on local `gpt-oss-20b`, not a change in the mission/runtime architecture.
- The Airbnb long-run bench and the local GPT-OSS harmony relay now both tolerate backend warmup instead of racing the first mission turn into `Connection refused`: the bench waits for `/ctox/telemetry` to report a healthy local backend before posting the initial mission prompt, and the harmony relay now retries transient upstream connect failures during the startup window instead of terminating Codex's response stream immediately.
- CTOX now has an explicit visible survival control plane instead of relying on hidden background control loops: queue pressure containment, timeout continuation, mission idle watchdog, and inbound safety boundaries persist governance events and are rendered back into each live chat turn so the agent can see which autonomous mechanisms exist and what they recently did.
- At the same time, CTOX stopped auto-driving normal work from heuristic control paths: automatic completion-review gating, automatic context-health repair enqueueing, automatic mission-loop repair enqueueing, blocked-review cron injection, and prose-based follow-up forcing were removed from the service path so non-survival heuristics are advisory unless the agent explicitly invokes them.
- The root `AGENTS.md` now codifies this as a standing project law for future threads: only explicit survival or safety mechanisms may act autonomously, every such mechanism must be visible to CTOX via persisted governance events, and new heuristic parser-driven control loops are treated as architectural regressions unless explicitly approved.
- CTOX now also has one central mechanism inventory in `governance`: survival, safety, advisory, and explicit correction paths are named in one place, the hard runtime blocker backoff is part of that visible survival set, and future threads can inspect the inventory directly instead of reconstructing correction logic from scattered modules.
